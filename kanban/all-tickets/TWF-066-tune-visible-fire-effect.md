# TWF-066: Tune Visible Fire Effect

## Goal

Make the live Timberborn fire effect clearly visible in normal gameplay camera angles without relying on alert text or status counters.

## Why

`TWF-040` through `TWF-046` prove the pooled native presentation path and coherent live loop, but the current bounded stimulus is too small and transient to produce a satisfying fire-effect recording. Fire needs its own visual tuning pass before release screenshots and Steam Workshop media.

The current design requires fire and smoke presentation to be field-based rather than per-tile. This ticket owns making the fire side of that rule convincing in live Timberborn.

## Requirements

- Use the existing pooled native effect path; do not create one entity per
  simulated fire cell.
- Follow `docs/DESIGN.md` section 17: cluster, blur, threshold, or otherwise
  aggregate neighboring fire intensity into larger coherent regions.
- Use compact deltas only to wake or bound visual regions; do not map one
  changed cell to one visible effect.
- Prefer Timberborn-native fire prefabs and conventions before custom art.
- Tune only presentation concerns such as scale, placement, lifetime, intensity
  thresholds, candidate selection, or pool reuse.
- Keep fire rules in the GPU simulation.
- Capture high-resolution screen recordings and screenshots where the fire
  effect itself is legible.
- Preserve command output, copied `Player.log`, artifact paths, and final QA
  lock state.
- Document accepted fire-effect evidence in `docs/TEST_PLAN.md`.

## Dependencies

- `TWF-065` provides the recording tool.

## Role

- Worker.
- Follow [worker.md](../roles/worker.md).

## Verification

- Run `git diff --check`.
- Run `dotnet test`.
- QA must capture high-resolution recording evidence plus `qa-readiness` or `status` evidence with active pooled fire effects, native prefab resolution, visible effects enabled, and zero presentation failures.

## Implementation Notes

- Start from the existing pooled native effect path and aggregate fire intensity into larger regions before changing pool size, placement, or thresholds.
- Keep any live-recording requirement blocked rather than satisfied by deterministic-only evidence if Timberborn cannot launch into a command-responsive loaded save.
