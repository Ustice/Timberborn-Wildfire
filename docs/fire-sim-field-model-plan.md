# FireSim Field Model One-Session Plan

This plan is for one focused implementation session. Do not turn it into kanban tickets or a sprint plan unless Jason asks for that later.

The goal is to replace the current threshold-based wind weighting in `FireSim.compute` with a small, readable field model:

- Heat spreads through a wind-shaped elliptical kernel.
- Smoke, steam, ash, and contamination remain separate atmospheric concepts.
- Memory usage stays close to the current packed-buffer design.
- The implementation uses red-green TDD: write failing tests first, then make the shader pass them.

## Ground Rules

- Work in a sibling worktree, not the main checkout.
- Start from a clean `FireSim.compute` on `main`.
- Remove or replace tests and snapshots that assert the old heat-spread implementation as part of setup.
- Use `bun` for TypeScript commands. Do not use `npm`.
- Keep the simulation core host-agnostic.
- Keep Timberborn as an adapter. Do not put fire rules in Timberborn systems.
- Use no more than five tuning levers for this work.

## Tuning Levers

Use exactly these five levers for the first pass. Do not add more without stopping and updating this plan.

- `HeatReach`: base heat influence radius.
- `WindStretch`: how strongly wind lengthens downwind heat reach and shortens upwind reach.
- `AtmosphericDrift`: how strongly smoke, steam, and ash prefer downwind movement.
- `AtmosphericDecay`: shared decay pace for transported atmospheric fields.
- `ContaminationCarry`: how much material contamination rides smoke and ash.

These can initially be shader constants. If runtime presets need them later, add them to `FireSimParameters` in a separate follow-up after the model is proven.

## Model Summary

Heat is a reaction driver. Smoke, steam, ash, and contamination are atmospheric outputs.

Use this mental model:

- Heat: short-range elliptical source kernel; affects ignition and burn pressure.
- Smoke: wind-carried field; generated by hot fuel and burning material.
- Steam: short-lived field; generated by heat plus water or moisture.
- Ash: persistent field/deposit; generated by fuel drop and smoke settling.
- Contamination: not an independent fluid; it rides smoke and ash.

Keep the current memory shape unless a test proves it cannot express the model:

- `CurrentCells` and `NextCells`: packed cell reaction state.
- `CurrentAtmosphericFields` and `NextAtmosphericFields`: packed atmospheric state.
- `CompanionFields`: read-only material metadata.
- `VisualFields`: presentation output only.

Do not add new full-size simulation buffers in this pass.

## Red-Green TDD Sequence

### 1. Red: No-Wind Heat Is Radial

Write a shader fixture test with one hot source in a flat grid and no wind.

Expected behavior:

- Equal-distance cells receive equal heat pressure.
- Opposite directions match.
- Heat does not reach beyond `HeatReach`.

Run the focused test and confirm it fails against the old implementation or missing new assertions.

### 2. Green: Implement Base Heat Kernel

In `src/Wildfire.Unity/FireSim.compute`, replace `WindWeightedNeighborHeat` and the old neighbor weighting with a base heat kernel:

```text
t = distance / HeatReach
weight = t < 1 ? (1 - t)^2 : 0
```

Keep this first pass no-wind only if needed. Make the radial test pass before adding wind.

### 3. Red: Wind Stretches Heat Downwind

Write a test with a single hot source and wind along one grid axis.

Expected behavior:

- Downwind cells receive more heat pressure than matching upwind cells.
- Crosswind cells sit between downwind and upwind.
- The source cell owns the outgoing shape; the target does not recompute its own wind preference.

### 4. Green: Add Polar Ellipse Reach

Implement the polar ellipse reach:

```text
headReach = HeatReach * (1 + WindStretch * windStrength)
backReach = HeatReach / (1 + WindStretch * windStrength)
e = (headReach - backReach) / (headReach + backReach)
a = (headReach + backReach) / 2
p = a * (1 - e^2)
reach(theta) = p / (1 - e * cos(theta))
```

Use:

```text
cos(theta) = dot(normalized source-to-target horizontal direction, normalized wind direction)
```

Then:

```text
t = distance / reach(theta)
weight = t < 1 ? (1 - t)^2 : 0
```

Make the wind-stretch test pass.

### 5. Red: Atmospheric Fields Use Transport, Not Heat Averaging

Write tests for atmospheric behavior using the existing packed atmospheric field shape.

Expected behavior:

- Smoke drifts downwind more strongly than upwind.
- Steam decays faster than smoke and is generated by heat plus water.
- Ash persists or decays more slowly than smoke.
- No test requires a new full-size buffer.

### 6. Green: Replace Nearest-Neighbor Atmospheric Max With Small Directional Transport

Update `BuildAtmosphericField` so smoke, steam, and ash use small directional transport rather than only four-neighbor max propagation.

Keep it cheap:

- Use a radius of `1` or `2` for atmospheric sampling.
- Use the same wind alignment helper as heat.
- Use `AtmosphericDrift` and `AtmosphericDecay`.
- Keep all atmospheric values packed into one `uint`.

Do not add per-field float buffers.

### 7. Red: Badwater Contamination Rides Smoke And Ash

Write tests for contaminated material or badwater-like material metadata.

Expected behavior:

- Contaminated source material emits contaminated smoke while hot or burning.
- Contaminated smoke can become contaminated ash/deposit when it decays or when fuel drops.
- Clean smoke does not create contamination.
- Contamination does not move without smoke or ash.

### 8. Green: Implement Contamination Carry

Use `CompanionContaminationBehavior` and `CompanionAshQuality` as the source classification.

Implement:

```text
sourceContamination =
  contaminated material or badwater behavior ? high :
  toxic ash quality ? high :
  normal material ? none
```

Then:

```text
smokeContamination = max(transportedSmokeContamination, smokeSourceContamination)
ashContamination = max(existingAshContamination, ashSourceContamination, depositedSmokeContamination)
```

Apply `ContaminationCarry` only when contamination moves with smoke or ash. Do not create a separate badwater vapor field in this pass.

## Implementation Notes

Prefer small helper functions in `FireSim.compute`:

- `HeatFalloff`
- `WindAlignment`
- `PolarEllipseReach`
- `HeatKernelWeight`
- `AtmosphericTransportWeight`

Remove old helpers once replaced:

- `WindWeightedNeighborHeat`
- Any old threshold-only heat bonus or penalty path that is no longer used.

Keep naming plain. The code should read like field math, not like a pile of magic constants.

## Done Conditions

The session is done when:

- The work was completed in `~/repos/wildfire-fire-sim-field-model`.
- Old spread-model assertions were removed or replaced.
- New tests prove no-wind radial heat, wind-stretched heat, atmospheric transport, and contamination carry.
- `FireSim.compute` no longer uses the old wind bonus/penalty heat model.
- The shader still uses packed atmospheric fields and does not add new full-size simulation buffers.
- There are no more than five tuning levers.
- The validation commands above pass, or failures are documented with the exact blocker and next command.

## Stop Conditions

Stop and ask Jason before continuing if:

- The implementation appears to need a sixth tuning lever.
- The packed atmospheric field cannot represent the required behavior.
- A new full-size buffer seems necessary.
- Fuel burn-down behavior becomes entangled with directional spread in a way that hides the fuel-duration proof.
- Live Timberborn validation becomes necessary to understand deterministic shader behavior.
